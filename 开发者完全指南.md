## 🔍 问题本质

### 错误信息

```
Error: The module 'better_sqlite3.node' was compiled against a different Node.js version
using NODE_MODULE_VERSION 127. This version requires NODE_MODULE_VERSION 136.
```

### 问题根源

```
你的系统 Node.js v22.18.0 (MODULE_VERSION 127)
    ↓ npm install/rebuild 使用系统 Node.js 编译
better_sqlite3.node (MODULE_VERSION 127)
    ↓ 但是...
Cursor 运行在 Electron 31.x 中
    ↓ Electron 内置 Node.js 20.x
需要 MODULE_VERSION 136
    ↓
❌ 版本不匹配 → 报错！
```

### 为什么反复出现？

因为常见的"解决方案"实际上不能解决问题：

```bash
npm install   # ❌ 用系统 Node.js 编译 → MODULE_VERSION 127
npm rebuild   # ❌ 还是用系统 Node.js → MODULE_VERSION 127
删除重装      # ❌ 还是会用系统 Node.js → MODULE_VERSION 127
```

**正确做法**：`npm run setup` ✅

## 📚 核心概念

### 1. 什么是原生模块（Native Module）

**原生模块** = C/C++ 代码编译成的 Node.js 模块

```
普通 JS 模块：
├─ index.js
└─ 纯 JavaScript 代码 → 跨平台、跨版本 ✅

原生模块：
├─ src/
│  └─ module.cc (C++ 代码)
├─ binding.gyp (编译配置)
└─ build/Release/
   └─ module.node (二进制文件) → 必须匹配特定版本 ❌
```

**为什么需要原生模块？**

```javascript
// 纯 JS 实现
const result = slowJsFunction(data);  // 100ms

// C++ 原生模块
const result = nativeFunction(data);   // 1ms ⚡ 快 100 倍！
```

**典型例子**：
- `better-sqlite3` - SQLite 数据库（你的项目）
- `node-sass` - SASS 编译器
- `sharp` - 图像处理
- `bcrypt` - 密码加密

### 2. NODE_MODULE_VERSION 是什么

**MODULE_VERSION** = Node.js 的 ABI（Application Binary Interface）版本号

| MODULE_VERSION | Node.js 版本 | Electron 版本 |
|----------------|--------------|---------------|
| 136 | 20.x, 23.x | 31.x, 39.x |
| **127** | **22.x** | **37.x, 38.x** |
| 120 | 21.x | 30.x |
| 115 | 20.x | 25-27.x |
| 108 | 18.x | 22-24.x |

**关键点**：
- 原生模块必须针对特定的 MODULE_VERSION 编译
- 不同 MODULE_VERSION 的二进制文件**不能混用**
- 这就像 Windows 的 `.dll` 文件必须匹配系统架构

### 3. VS Code/Cursor 扩展的运行环境

```
┌─────────────────────────────────┐
│  你的 Windows 系统               │
│                                  │
│  系统 Node.js v22.18.0          │ ← npm 命令使用
│  (MODULE_VERSION 127)           │
│                                  │
│  ┌─────────────────────────┐    │
│  │ Cursor/VS Code          │    │
│  │  └─ Electron 37.7.0     │    │
│  │     └─ Node.js 22.20.0  │    │ ← 扩展实际运行环境
│  │     (VERSION 127)       │    │
│  └─────────────────────────┘    │
│                                  │
│  你的扩展代码                     │
│  └─ better_sqlite3.node         │ ← 必须匹配 Electron 的版本
└─────────────────────────────────┘
```

**重要理解**：
- VS Code/Cursor 是基于 **Electron** 构建的
- **Electron = Chromium + Node.js**
- 你的扩展运行在 Electron 内置的 Node.js 中
- **不是**运行在系统 Node.js 中！

### 4. 三重兼容性问题

#### 问题 1：MODULE_VERSION 不兼容

```
开发环境：Electron 37 (MODULE_VERSION 127)
用户环境：Electron 31 (MODULE_VERSION 136)
    ↓
❌ 不兼容
```

#### 问题 2：平台不兼容

```
Windows .node ≠ macOS .dylib ≠ Linux .so
    ↓
❌ 不能跨平台使用
```

#### 问题 3：架构不兼容

```
x64 (Intel/AMD) ≠ arm64 (Apple Silicon)
    ↓
❌ 不能跨架构使用
```

- 它们使用**相同的 MODULE_VERSION 127**
- 只要 MODULE_VERSION 相同，二进制就兼容

## 👥 用户安装兼容性

### 核心挑战

当你打包 `.vsix` 文件时，只包含**一个版本**的二进制文件：

```
你的开发环境：
├─ Windows x64 + Electron 37 (MODULE_VERSION 127)
├─ 打包 .vsix
└─ 包含 better_sqlite3.node (MODULE_VERSION 127, Windows x64)

用户 A：Windows x64 + Electron 37 → ✅ 完美兼容
用户 B：Windows x64 + Electron 31 → ❌ MODULE_VERSION 不匹配
用户 C：macOS + Electron 37 → ❌ 平台不匹配
用户 D：macOS M1 + Electron 37 → ❌ 平台+架构不匹配
```

### 兼容性矩阵

| 你的环境 | 用户环境 | 结果 |
|----------|----------|------|
| Win x64, E37 | Win x64, E37 | ✅ 完美 |
| Win x64, E37 | Win x64, E31 | ❌ MODULE_VERSION 不匹配 |
| Win x64, E37 | macOS x64, E37 | ❌ 平台不匹配 |
| Win x64, E37 | macOS arm64, E37 | ❌ 平台+架构不匹配 |
| Win x64, E37 | Linux x64, E37 | ❌ 平台不匹配 |

**估计**：
- ✅ 直接兼容：20-30% 用户
- ❌ 需要适配：70-80% 用户

### 为什么 SQLite 是 C++ 有影响？

**better-sqlite3 的架构**：

```
better-sqlite3/
├─ lib/              ← JavaScript 包装代码（跨版本兼容 ✅）
├─ src/              ← C++ 源代码（需要编译 ❌）
│  ├─ database.cc
│  └─ statement.cc
└─ build/Release/
   └─ better_sqlite3.node  ← 编译后的二进制（不跨平台 ❌）
```

**为什么用 C++？**

```javascript
// 纯 JS 实现 SQLite（如 sql.js）
const db = new SQL.Database();
db.run("SELECT * FROM users");
// 性能：~100 queries/sec

// C++ 原生实现（better-sqlite3）
const db = require('better-sqlite3')('db.sqlite');
db.prepare("SELECT * FROM users").all();
// 性能：~10,000 queries/sec ⚡
```

**代价**：二进制文件必须针对特定环境编译。

### 用户安装解决方案（已实施）

#### 运行时动态下载（Runtime Download）

我已经为你实现了自动适配系统：

```typescript
// src/sqlite-loader.ts
export class SqliteLoader {
    async ensureLoaded() {
        // 1. 检测当前二进制是否可用
        if (await this.testBinary()) {
            return true; // 已有兼容版本 ✅
        }
        
        // 2. 检测 Electron 版本
        const electronVersion = process.versions.electron;
        
        // 3. 下载匹配的预编译二进制
        await this.downloadBinary(electronVersion);
        
        // 4. 验证是否成功
        return await this.testBinary();
    }
}

// src/extension.ts
export async function activate(context) {
    // 激活时自动检查并下载
    const sqliteLoader = new SqliteLoader(context.extensionPath);
    await sqliteLoader.ensureLoaded();
    
    // 然后启动扩展功能...
}
```

**工作流程**：

```
用户安装 .vsix
    ↓
VS Code/Cursor 激活扩展
    ↓
SqliteLoader.ensureLoaded()
    ├─ 检测：二进制文件存在且兼容？
    │   ├─ 是 ✅ → 立即使用（< 1秒）
    │   └─ 否 ❌ → 自动下载
    │       ↓
    │       显示通知："Setting up database module..."
    │       ↓
    │       检测环境（Electron 版本、平台、架构）
    │       ↓
    │       下载匹配的预编译版本（~2MB，2-5秒）
    │       ↓
    │       验证成功 ✅
    │       ↓
    │       显示："✅ Setup complete!"
    └─ 扩展开始工作
```

## ✨ 总结

### 核心要点

1. **原生模块必须针对特定版本编译**
2. **Cursor 扩展运行在 Electron 的 Node.js 中**
3. **系统 Node.js ≠ Electron Node.js**
4. **MODULE_VERSION 才是关键，不是具体版本号**

### 问题根源

```
npm install/rebuild → 使用系统 Node.js → 版本可能不对 ❌
```

### 正确做法

```
npm run setup → 下载 Electron 对应版本 → 完美匹配 ✅
```

### 一句话记住

**不要用系统 Node.js 编译原生模块，要用 Cursor 对应的 Electron 版本！**
